# Evidence Index

This folder contains deterministic evidence for the completed platform milestones. Evidence is generated from synthetic NexaFlow data and local scripts. It is compact by design and does not duplicate large runtime outputs.

## `milestone-3`

Purpose: ingestion and data-quality evidence.

Key files: `ingestion-manifest.json`, `quality-report.json`, `quality-report.md`, `lineage.json`, `run-metrics.json`.

Generated by: `make verify-ingestion-evidence`.

Proves: source validation, checksums, accepted/quarantined handling, quality reporting, and lineage.

Caveats: synthetic sample only.

## `milestone-4`

Purpose: governed funnel analytics evidence.

Key files: `funnel-summary.csv`, `funnel-stage-metrics.csv`, `funnel-dropoff-analysis.csv`, `analysis-manifest.json`.

Generated by: `make verify-funnel-evidence`.

Proves: deterministic funnel conversion, drop-off, diagnostics, and lineage.

Caveats: descriptive synthetic journeys.

## `milestone-5`

Purpose: retention and cohort evidence.

Key files: `cohort-summary.csv`, `retention-matrix.csv`, `retention-long.csv`, `resurrection-analysis.csv`.

Generated by: `make verify-retention-evidence`.

Proves: deterministic cohort retention, censoring, lifecycle, and resurrection analysis.

Caveats: cohort sizes are synthetic and small.

## `milestone-6`

Purpose: churn prediction evidence.

Key files: `evaluation-metrics.json`, `threshold-analysis.csv`, `feature-importance.csv`, `model-card.md`, `model-lineage.json`.

Generated by: `make verify-churn-evidence`.

Proves: leakage-safe model training, chronological splits, evaluation, and model documentation.

Caveats: risk demonstration only; not for automated adverse decisions.

## `milestone-7`

Purpose: user segmentation evidence.

Key files: `segment-profiles.csv`, `cluster-candidate-metrics.csv`, `cluster-stability.csv`, `pca-explained-variance.csv`.

Generated by: `make verify-segmentation-evidence`.

Proves: deterministic segment profiles, stability, naming, and governance.

Caveats: segment labels are interpretive, not causal.

## `milestone-8`

Purpose: recommendation baseline evidence.

Key files: `model-comparison.csv`, `offline-metrics.json`, `metrics-by-k.csv`, `catalogue-coverage.csv`, `recommendation-card.md`.

Generated by: `make verify-recommendation-evidence`.

Proves: deterministic offline recommendation comparison, ranking metrics, coverage, novelty, and diversity.

Caveats: offline metrics are not user-facing probabilities.

## `milestone-9`

Purpose: experiment analysis evidence.

Key files: `decision-summary.csv`, `metric-results.csv`, `guardrail-results.csv`, `sample-ratio-mismatch.csv`, `power-analysis.csv`.

Generated by: `make verify-experiment-evidence`.

Proves: fixed-window experiment analysis with SRM, guardrails, statistical testing, power, and decision rules.

Caveats: synthetic experiment sizes can be underpowered.

## `milestone-10`

Purpose: deterministic product insight assistant evidence.

Key files: `grounded-insights.json`, `executive-product-insight-report.md`, `risk-and-caveat-register.csv`, `assistant-run-manifest.json`.

Generated by: `make verify-product-insight-evidence`.

Proves: grounded insight generation, governance checks, lineage, and no-live-call behavior.

Caveats: default assistant is deterministic and local.

## `milestone-11`

Purpose: Power BI-ready reporting and semantic layer evidence.

Key files: `semantic-model.json`, `metric-dictionary.csv`, `dashboard-pages.json`, `visual-specifications.json`, reporting fact and dimension CSVs.

Generated by: `make verify-reporting-evidence`.

Proves: analytics-to-reporting handoff, semantic documentation, metric dictionary, dashboard specs, lineage, and checksums.

Caveats: no `.pbix` file or Power BI deployment is created.

## `milestone-12`

Purpose: final Azure architecture and portfolio readiness evidence.

Key files: `final-platform-summary.md`, `azure-service-mapping.json`, `repo-health-summary.json`, `final-lineage-overview.json`, `milestone-12-manifest.json`.

Generated by: `make verify-final-evidence`.

Proves: final architecture coverage, local-to-Azure mapping, repo health, portfolio readiness, and deployment boundaries.

Caveats: deployment readiness is documented only; no live Azure resources are provisioned.
